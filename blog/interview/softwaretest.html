# 🚀软件测试

# 软件测试按照开发阶段分为几类？你认为最重要的是哪个？

按照开发阶段可以分为 5 类，分别是：

  • 单元测试
  • 集成测试
  • 系统测试
  • 验收测试
  • 回归测试

我认为最重要的是 系统测试，理由如下：

首先在进行 系统测试 的时候，单元测试和集成测试已经完成了。这就能保证构成软件的最小功能正确，并且能够保证模块与模块之间的接口也都OK。

其次，开发软件就是为了能满足用户提出的需求文档的所有功能，提供一个可交付的软件。那么系统测试就是对软件的全部功能进行测试，里边覆盖系统的所有联合部件,是针对整个软件系统进行完成整的、全面的测试的。除此之外它还能验证软件实现的功能是否能满足需求文档规范的功能。由此看来它的作用很大，因此我认为在这几个测试中，系统测试很重要。

# α 测试 和 β 测试是什么？两者有什么区别？

# 概念

α 测试：是用户来进行测试的，不能有开发人员或者测试人员来完成。在测试的时候，用户会被邀请到公司中，在开发环境下进行测试。

β 测试：是验收测试，它要求软件的最终用户们，在一个或者多个场所下对软件进行测试。

# 两者区别

从测试人数、时间集中性、用户数量上

  • α 测试 环境受开发方控制，测试的人数相对较少，时间较为几种
  • β 测试 环境不受开发方控制，测试的人数相对较多，时间不集中

从测试的时间先后顺序上：

  • α 测试 先于 β 测试

从测试场所上看：

  • α 测试 是在公司内部，处于开发环境中进行测试的
  • β 测试 是在多个场所下进行测试的

# 谈一谈自动化测试和非自动化测试的优缺点

# 概念：

自动化测试：将人手写测试的全过程，通过代码的编写，让机器去执行。进而解放了人的双手，提高了效率。

非自动化测试：手工测试，编写测试用例，一个一个去进行测试，并观察最终的结果。

# 自动化测试优缺点

优点：

  • 提升了工作效率：机器代替人工进行测试。只要脚本设计的可靠性高，那么机器在执行的时候不管是运算、输入以及得出结果都会比人工快的多
  • 让回归测试更方便：品在发布之后，并不是一成不变的。它会根据用户的使用，改进或者增添新的功能，新功能的引入势必会牵扯到回归测试。因为自动化在回归测试的上一次已经对大部分功能测试编写好了脚本，后边回归测试时只需要简单的改一下脚本就好，让回归变得更方便。
  • 能腾出更多的资源：能够提升效率，在编写好自动化测试脚本之后，测试人员直接开启脚本自测即可。
  • 抗错误能力好：手工测试，只要是人来进行的事情就有可能出错。但是机器执行就避开了这一点，只要你测试脚本写的没问题，机器从来不会因为工作时间长或者工作疲惫而造成错误。

缺点：

  • 没有探索性，一根筋：由于是机器执行，根本不会对测试进行改进。人们设置啥，他就执行啥，在实际测试过程中，可能人在测试的时候突然想到了更好的测试方法，或者出现了新的测试点。但是自动化测试时完全做不到。
  • 没办法代替手工测试：
  • 会制约软件开发：
  • 对测试质量的依赖性很大：依赖于测试人员编写的自动化脚本，因此脚本的质量直接决定了测试的最终成果。

# 手工测试的优缺点

优点：

  • 能进行探索性测试：能够在测试的过程中发现新的点子
  • 能及时止损：一旦在刚开始发现错误就会进行处理，不想自动化那样，，一股脑的进行

缺点：

  • 执行效率低：
  • 不能模拟大数据的测试
  • 出错率比较高

# 集成测试和系统测试的区别是什么？两者分别用于什么场景？

# 区别

首先来看一下 软件测试 V 模型

  • 测试用例的个数不同

集成测试：相比于系统测试，测试点更细。所以 集成测试的测试用例要多于系统测试。

  • 测试用例的编写顺序不同

由上图可以看出，首先是先进行需求分析与系统然后再进行概要设计的。因此对应右边的软件测试流程就是先进行系统测试然后再进行集成测试。

  • 用例的粒度不同

集成测试相对于系统测试来说测试点要更多，所以系统测试的用例设计更详细。

  • 执行的顺序不同

先进行系统测试然后再进行集成测试

# 应用场景

集成测试：在单元测试完成之后，主要做的事情就是 对各个模块直接接口的测试。它可以是整个产品的集成测试，也可以是大规模的集成测试。一般采用 黑盒 + 白盒 的手段来进行。

系统测试：在单元测试和集成测试完成之后，针对软件的整个功能模块进行测试，要模拟用户实际应用场景，看开发完成的软件是否符合需求文档规范。

# 谈一下什么是静态测试？什么是动态测试？

静和动就是不运行代码和运行代码。

静态测试：在不运行代码的情况下，让人分析代码的语法结构、代码质量、文档测试等。

动态测试：在程序运行起来后，分析运行结果、性能、接口、内存、覆盖等等。
